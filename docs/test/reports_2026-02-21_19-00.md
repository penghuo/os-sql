### Analysis

**All 1864 failures are pre-existing serialization issues** unrelated to this PR's changes. The three dominant errors are:

1. **CalciteLogicalIndexScan(RelInput) missing constructor** (~1280 failures): The `RelNodeSerializer` cannot deserialize `CalciteLogicalIndexScan` because it lacks a `(RelInput)` constructor required by Calcite's JSON serde. This is a pre-existing gap in the serialization layer.

2. **LogicalSystemLimit.SystemLimitType not serializable** (~200 failures): `LogicalSystemLimit` uses a custom enum `SystemLimitType` which `RelJson.toJson()` cannot serialize. This was visible in the build output stack trace.

3. **SqlTypeName.BIGINT enum not serializable** (~40 failures): Another serialization gap where Calcite `SqlTypeName` enum values are not handled by the extended JSON writer.

**None of these failures are caused by the coordinator plan fix.** They are all in the shard plan serialization path (`RelNodeSerializer.serialize()` -> `executeExchange()`) which runs before `buildResponse()` is ever called. The changes in this PR (PlanSplitter MergeAggregateExchange fix, two-path buildResponse, ExchangeLeaf convention) are not exercised by these failing tests because execution fails at the serialization step.
