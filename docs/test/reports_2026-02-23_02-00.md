---

## Changes Under Test

1. **Narrowed `containsCustomUDFCalls` to `containsRelevanceFunctions`** in `PlanSplitter.java`: Only rejects plans containing relevance functions (match, query_string, multi_match, etc.) from DQE. Regular UDFs (date/time, JSON, string, math, array, etc.) are now allowed through the scatter-gather path.

2. **Retained Interpreter for shard-side execution**: Instead of code-generation (RelRunner), UDF-containing plans execute via the Calcite Interpreter on shards, avoiding the Volcano planner re-optimization that caused NPE and CannotPlanException in Runs 8a/8b.

3. **`toOp()` fix**: Corrected UDF operator conversion to ensure proper handling in the Interpreter path.

### Key Findings

1. **Interpreter approach works**: Recovering from 115 to 1083 passing tests validates the strategy of keeping the Interpreter for shard-side UDF execution instead of RelRunner + Volcano planner re-optimization.

2. **+380 tests above strict baseline**: Run 9 passes 1083 tests vs the strict mode baseline of 703, a **54.1% improvement** over the baseline. This means the DQE path is handling more queries correctly than strict-mode-off.

3. **-612 tests vs Run 7**: Still 612 tests below the Run 7 peak (1695). The gap is primarily due to:
   - NullPointerException (121 tests) -- residual NPEs needing investigation
   - Cannot translate / Janino CompileException (170 tests) -- Interpreter limitations with certain expressions (window functions, complex aggregates)
   - UnsupportedOperationException (127 tests) -- lambda serialization and other unsupported operations
   - Relevance function failures (106 tests) -- expected; these are correctly gated
   - Subquery failures (33 tests) -- EXISTS and scalar subqueries not in DQE scope
   - Assertion mismatches (66 tests) -- result correctness issues

4. **Relevance function gating confirmed**: All 7 relevance function test classes (Match, MatchPhrase, MatchPhrasePrefix, MatchBoolPrefix, MultiMatch, QueryString, SimpleQueryString, RelevanceFunction) correctly fail, confirming `containsRelevanceFunctions` gate works as intended.

### Recommended Next Steps

1. **Investigate NullPointerException (121 tests)**: These are server-side NPEs, not the same ProjectIndexScanRule NPE from Run 8a. Root cause likely in shard-side Interpreter execution or expression evaluation.

2. **Address Interpreter translation gaps (170 tests)**: Window functions (`MIN() OVER()`, `MAX() OVER()`) and certain aggregate expressions fail with "cannot translate" or CompileException. May require custom Interpreter nodes or expression rewrites.

3. **Fix lambda serialization (127 tests)**: Aggregate reduce lambdas cannot be serialized by `ExtendedRelJson.toJson()`. Need custom serialization support for lambda/closure RexNodes.

4. **Investigate assertion mismatches (66 tests)**: Result correctness issues where queries execute but produce wrong results.

5. **Subqueries (33 tests)**: EXISTS and scalar subqueries are a known DQE limitation -- may need separate handling or explicit skip.
