## Changes Under Test

### Run 8a (initial UDF fix)

1. **Replace `containsCustomUDFCalls` with `containsRelevanceFunctions`** in `PlanSplitter.java`: Narrows the DQE gate to only reject plans containing relevance functions (match, query_string, etc.). Regular UDFs (date/time, JSON, string, math, etc.) are now allowed through the DQE scatter-gather path.

2. **ShardCalciteRuntime changes**: Updated shard-side execution to use code-generation (RelRunner) for plans containing UDFs instead of the Interpreter.

3. **OpenSearchExecutionEngine changes**: Adjusted execution engine to support the new UDF execution path.

### Run 8b (NPE hotfix)

4. **BindableTableScan shuttle** in `ShardCalciteRuntime.java`: Added a `RelHomogeneousShuttle` that replaces `CalciteLogicalIndexScan` nodes with `BindableTableScan` before calling `RelRunner.prepareStatement()`, preventing OpenSearch-specific pushdown rules (e.g., `ProjectIndexScanRule`) from firing during shard-side Volcano planner re-optimization.

### Root Cause Analysis

**Run 8a primary issue (fixed in 8b)**: NullPointerException in `ProjectIndexScanRule.apply()` at line 54. The shard-side `RelRunner.prepareStatement()` re-optimizes via Volcano planner, firing OpenSearch-specific pushdown rules that expect metadata not present in deserialized plans.

**Run 8b primary issue (new)**: The BindableTableScan shuttle fixes the NPE but introduces a **convention mismatch**. `BindableTableScan` has convention `BINDABLE`, but the `RelRunner` requests `ENUMERABLE` convention. The Volcano planner has no rule to convert `LogicalProject[NONE -> ENUMERABLE]` when the leaf is `BINDABLE`. This affects 1457 tests (79.7% of failures).

**Persistent issues (both runs)**:
- **UnsupportedOperationException (126 tests)**: Lambda expressions in aggregate reduce functions (`(acc, x) -> +(acc, CHAR_LENGTH(x))`) cannot be serialized by `ExtendedRelJson.toJson()`.
- **AssertionError (166-188 tests)**: Backend assertion failures, type mismatches.

### Verdict

**The fix remains broken after the NPE hotfix.** The BindableTableScan shuttle successfully eliminated the NPE (1467 -> 23), but it introduced an equally fatal **CannotPlanException** (36 -> 1457). The core problem: `BindableTableScan` has `BINDABLE` convention, but `RelRunner.prepareStatement()` requests `ENUMERABLE` convention, and there are no Calcite rules registered to bridge this gap.

### Recommended Next Steps

1. **Fix convention mismatch**: Either:
   - (a) Use `EnumerableTableScan` instead of `BindableTableScan` in the shuttle, so the leaf has ENUMERABLE convention matching what RelRunner expects, OR
   - (b) Register `BindableToEnumerableConverterRule` in the shard-side planner so Calcite can bridge BINDABLE -> ENUMERABLE, OR
   - (c) Skip the Volcano planner entirely on the shard side -- since the plan is already optimized by the coordinator, the shard should just execute it without re-optimization.

2. **Fix lambda serialization (126 tests)**: Add support for serializing aggregate reduce lambdas in `ExtendedRelJson.toJson()`.

3. **Fix AssertionError growth (188 tests)**: Investigate why backend assertion failures increased from 32 to 188.

4. **Consider approach (c)**: The fundamental design issue is that `RelRunner.prepareStatement()` re-optimizes the plan through Volcano, which fires rules that don't apply on the shard side. If the shard directly executed the plan without re-optimization (e.g., via Interpreter or direct Enumerable compilation), most of these issues would be avoided.
