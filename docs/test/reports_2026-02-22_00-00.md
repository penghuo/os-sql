## Changes Under Test

Three P1 correctness fixes applied in this run:

1. **Optimization ordering (Task #1/#7)**: `CalciteToolsHelper.optimize(rel, context)` is now called before `PlanSplitter.split(rel)` in the DQE path of `OpenSearchExecutionEngine.execute()`. Previously, the split happened on the pre-optimization plan, meaning pushdown rules had not yet been applied.

2. **Scan deserialization (Task #2/#8)**: Investigation and fix for `DSLScan(RelInput)` constructor which created an empty `PushDownContext`. After optimization, scan nodes carry pushed filters/projects/sorts/limits that must survive serialization.

3. **ShardScannableTable pushdown context (Task #3/#9)**: Investigation and fix for `ShardScannableTable.scan()` in `ShardCalciteRuntime.java` which created an empty `PushDownContext`. For the Interpreter path, the scan's pushdown state matters for the fast path where the scan is the only node.

### Verdict

The three P1 fixes (optimization ordering, scan deserialization, ShardScannableTable pushdown context) produced **no regressions** in either unit tests or integration tests. Unit tests remain at 1501/1504 passed (3 pre-existing skips). Integration tests remain at 339/2008 passed.

**The P1 fixes are foundational correctness changes** -- they ensure the DQE path optimizes before splitting, preserves pushdown state through serialization, and handles the Interpreter scan wrapper correctly. These fixes do not directly resolve the remaining 1604 integration test failures, which are execution-layer issues:

1. **ExchangeLeaf reflection access (588, 36.7%)**: Highest priority. The Interpreter encounters `PlanSplitter$ExchangeLeaf` inner class nodes and cannot access them via reflection. These nodes should be stripped or replaced before Interpreter execution.

2. **Object[] type coercion (246, 15.3%)**: The Interpreter returns rows as `Object[]` arrays but downstream Calcite operators expect scalar values. A row-unwrapping adapter is needed.

3. **Missing date/time Calcite methods (228, 14.2%)**: Custom `date()`, `time()`, `datetime()`, `week()` functions are not registered in the Interpreter's method resolution.

4. **NullPointerException (106, 6.6%)**: Various null dereferences during shard-side execution need investigation.

5. **Relevance search not pushed down (105, 6.5%)**: Full-text search functions (match, match_phrase, etc.) cannot execute in the Interpreter and must be pushed down to OpenSearch.

6. **Window function translation (94, 5.9%)**: ROW_NUMBER, MAX/SUM/COUNT OVER, etc. are not translatable in Interpreter mode.

The P1 fixes ensure the pipeline is correct; the remaining failures are the "next layer" of implementation work needed for the Interpreter execution path.
