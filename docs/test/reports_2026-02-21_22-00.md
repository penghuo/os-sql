### Verdict

The `cannot convert table to enumerable` error is now **fully resolved**. This was the dominant failure (1699 out of 1863 failures in Run 3, 91%). The fix enables shard-side execution through Calcite's Interpreter with a ScannableTableAdapter that wraps the RelOptTable.

**Passed tests increased from 80 to 339 (+259, +324% improvement).**

The remaining 1604 failures fall into several new categories that represent the "next layer" of issues now that the execution path is functioning:

1. **ExchangeLeaf access errors (576)**: Reflection/module access issue when the Interpreter encounters `PlanSplitter$ExchangeLeaf` nodes. Likely needs the ExchangeLeaf to be stripped or replaced before Interpreter execution.

2. **Object[] type coercion (452)**: The Interpreter returns rows as `Object[]` arrays, but downstream operators expect scalar values (Number, String, Comparable). This is a row-unwrapping issue in the Interpreter execution path.

3. **Missing Calcite built-in methods (446)**: Custom date/time functions (`date()`, `time()`, `datetime()`) are not available in the Interpreter's runtime. These are custom OpenSearch SQL functions that need to be registered in the Interpreter's function table.

4. **Relevance functions not pushed down (208)**: Full-text search functions (match, match_phrase, etc.) cannot run in the Interpreter and must be pushed down to OpenSearch. The shard-side planner needs to handle these differently.

5. **Window function translation (76)**: ROW_NUMBER, MAX OVER, etc. cannot be translated by the Interpreter. These may need special handling or a different execution strategy.

These are all execution-layer issues, not serialization issues. The serde and plan-conversion pipeline is now working end-to-end.
