### 2.2 Failure Analysis

**ORDER BY tie-breaking (9 queries: Q19, Q24, Q25, Q31, Q32, Q33, Q39, Q40, Q41)**

When multiple rows share the same ORDER BY sort value, DQE and ClickHouse select different rows for the LIMIT. Both results are valid SQL — the output is non-deterministic when ties exist.

Example (Q24): `SELECT * FROM hits WHERE URL LIKE '%google%' ORDER BY EventTime LIMIT 10`
- Multiple rows have identical `EventTime` values
- CH picks rows by internal storage order; DQE picks by shard merge order
- All returned rows are valid members of the tie-set

**Float precision (Q4)**

`SELECT AVG(UserID) FROM hits` — `AVG()` on BIGINT values produces different floating-point rounding between CH and DQE due to accumulator precision differences.

**Non-deterministic (Q18)**

`SELECT UserID, SearchPhrase, COUNT(*) FROM hits GROUP BY UserID, SearchPhrase LIMIT 10` — no ORDER BY, so any 10 groups are valid output.

## 3. Architecture & Performance Analysis

### 3.1 Execution Architecture

```
Client → HTTP → DQE Coordinator
                    │
                    ├─ PlanFragmenter: decompose query into shard plan + coordinator plan
                    │
                    ├─ Dispatch to 8 shards (parallel)
                    │   ├─ Shard 0: DocValues scan → fused GROUP BY → partial result
                    │   ├─ Shard 1: DocValues scan → fused GROUP BY → partial result
                    │   ├─ ...
                    │   └─ Shard 7: DocValues scan → fused GROUP BY → partial result
                    │
                    ├─ Coordinator Merge: merge partial results → final result
                    │
                    └─ Response formatting → JSON → HTTP response
```

### 3.2 Time Breakdown by Component

Typical query execution breakdown (using Q9 as example):

| Component | Time | % | Description |
|-----------|-----:|--:|------------|
| HTTP overhead | 10ms | 33% | Request parsing, response serialization, network |
| Shard execution | 9ms | 30% | DocValues scan + hash aggregation (125K docs, parallel across 8 shards) |
| Coordinator merge | 8ms | 27% | Union HashSets across shards + top-K selection |
| Plan + dispatch | 3ms | 10% | Query plan cache lookup + shard dispatch |
| **Total** | **30ms** | | |

### 3.3 Key Optimizations Applied

| Optimization | Queries Affected | Improvement | Technique |
|-------------|-----------------|-------------|-----------|
| Native VARCHAR COUNT(DISTINCT) with HashSets | Q14 | 82ms → 35ms | Collect-then-sequential-scan + ordinal-indexed accumulators + top-K pruning at coordinator |
| Lazy merge + top-K pruning for HashSet merge | Q9 | 60ms → 29ms | Only merge groups that could be in top-K; take ownership of largest shard's set (zero-copy) |
| Native mixed COUNT(DISTINCT) with HashSets | Q10 | 53ms → 47ms | GROUP BY key0 only with per-group HashSets + SUM/COUNT accumulators; reduces coordinator merge from 200K → 3,200 rows |
| TopN inflation for COUNT(*)-only GROUP BY | Q40 | 580ms → 67ms | Detect COUNT(*)-only aggs and skip numShards*2 inflation |
| Inline `length()` with ordinal pre-computation | Q28 | 0 (wrong) → 30ms (correct) | Pre-compute ordinal→length map for O(1) per-doc lookup |
| REGEXP_REPLACE backreference fix | Q29 | 0 rows → correct | Convert SQL `\N` → Java `$N` in regex replacement |
| SELECT literal column handling | Q35 | crash → 14ms | Filter literal GROUP BY keys + inject in formatResponse |
| Always build shard+coordinator plan | all | Enables fast paths for single-shard indices | Remove multiShard guard in PlanFragmenter |

### 3.4 Why DQE is Faster Than ClickHouse on 17 Queries

DQE outperforms ClickHouse (ratio < 0.75x) on queries that benefit from:

1. **Lucene's inverted index for WHERE filters**: Queries with selective WHERE clauses (e.g., `CounterID = 62`) use Lucene's posting lists to skip non-matching docs entirely. ClickHouse must scan column blocks sequentially.

2. **SortedSetDocValues ordinal caching**: For VARCHAR GROUP BY, DQE maps each unique string to an integer ordinal once, then uses the ordinal for hash-free array-indexed accumulation. This is faster than ClickHouse's string hashing.

3. **TopN pre-filtering at shard level**: For ORDER BY + LIMIT queries, each shard caps its output to 1,000 groups (inflated limit). ClickHouse processes all groups before sorting.

4. **Capped GROUP BY insertion**: For GROUP BY + LIMIT without ORDER BY, shards stop after N groups. ClickHouse computes all groups.

### 3.5 Why DQE is Slower on Q10 and Q28

**Q10 (3.39x)**: `GROUP BY RegionID, SUM(AdvEngineID), COUNT(*), AVG(ResolutionWidth), COUNT(DISTINCT UserID)`

| Component | DQE | ClickHouse | Gap |
|-----------|----:|----------:|-----|
| Shard scan (parallel) | 20ms | — | 8 shards × 125K docs each |
| Coordinator merge | 10ms | — | Union 3,200 HashSet entries |
| HTTP + dispatch | 13ms | — | Overhead not present in CH |
| Single-node execution | — | 18ms | 1 pass, multi-threaded, vectorized |

ClickHouse processes COUNT(DISTINCT) natively in a single pass with per-group hash sets and multi-threaded vectorized column processing. DQE distributes across 8 shards which adds merge + HTTP overhead.

**Q28 (2.31x)**: `AVG(length(URL))` with GROUP BY

The `length(URL)` computation requires reading VARCHAR DocValues (`SortedSetDocValues.advanceExact()`) per document, which is slower than numeric DocValues. Even with ordinal→length pre-computation, the per-doc ordinal lookup adds overhead compared to a pure numeric scan.

## 4. Shard Count Analysis

### 4.1 Experiment: 1 Shard vs 8 Shards

We tested the same queries on a 1-shard index to understand the impact of shard-level parallelism:

| Metric | 8 Shards | 1 Shard |
|--------|---------|---------|
| Within 2x CH | 41/43 | 21/43 |
| Aggregate ratio | 0.75x | 9.78x |
| Correctness | 32/43 | 35/43 |

### 4.2 Why 8 Shards is Faster (Q9 Example)

```sql
SELECT RegionID, COUNT(DISTINCT UserID) FROM hits_1m
GROUP BY RegionID ORDER BY COUNT(DISTINCT UserID) DESC LIMIT 10
```

```
                        8 SHARDS (30ms)              1 SHARD (58ms)
                        ───────────────              ──────────────
Docs per shard:         125,000                      1,000,000
Parallelism:            8-way                        none

Shard scan:             9ms (125K docs)              58ms (1M docs)
                        × 8 parallel = 9ms wall      × 1 = 58ms wall
Coordinator merge:      8ms (union 8×400 HashSets)   1ms (passthrough)
HTTP + dispatch:        13ms                         9ms
                        ─────                        ─────
Total:                  30ms                         68ms
```

The key equation: `scan(1M docs) = 8 × scan(125K docs)` but parallel execution makes `wall_time(8 shards) = scan(125K)` not `scan(1M)`. The merge overhead (+8ms) is far less than the parallelism benefit (-49ms).

### 4.3 Why 1 Shard Has Better Correctness

With 1 shard, there is no cross-shard merge. The sort order is deterministic (single Lucene segment order), eliminating tie-breaking differences. This is why 3 additional queries pass (Q25, Q31, Q39).

### 4.4 Comparison with ClickHouse Architecture

| Aspect | DQE (8 shards) | ClickHouse |
|--------|---------------|------------|
| Parallelism | Inter-shard (8 shards in parallel) | Intra-query (N threads within 1 table) |
| Scan overhead | 125K docs/shard | 1M docs shared across threads |
| Merge overhead | Coordinator merge across shards | In-process merge (no serialization) |
| HTTP overhead | ~10ms per query | ~8ms (TCP, no REST) |
| COUNT(DISTINCT) | Decompose → HashSets → union | Native single-pass with per-group sets |
| Hash table | Primitive `long[]` open-addressing | Specialized 30+ variants with CRC-32C |
| Storage | Lucene DocValues (columnar per-segment) | MergeTree (columnar with sparse index) |
