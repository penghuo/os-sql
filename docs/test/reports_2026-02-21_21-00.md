### New Dominant Error

The serde layer now works end-to-end. Plans serialize, deserialize, and the plan root is correctly walked to find the `Scannable` leaf. However, shard-side execution fails at the **Calcite planner/optimizer stage**:

```
CalciteUnsupportedException: cannot convert table RelOptTableImpl{...} to enumerable
```

This error occurs because after deserialization, the shard plan is re-planned through Calcite's `VolcanoPlanner`, which tries to convert the `OpenSearchIndex`-backed table to an `EnumerableTableScan`. The table does not implement `ScannableTable` or `QueryableTable` in a way that Calcite's enumerable convention can handle directly. This is **not** a serialization issue -- it is an execution-layer gap.

### Verdict

All serialization/deserialization issues from the original 3 bugs plus the 2 follow-up issues (plan-root not Scannable, UDF deserialization) are now **fully resolved**. The serde layer works end-to-end.

The remaining 1863 failures are dominated by a single new issue: **`cannot convert table to enumerable` (1699 failures, 91%)**. This is an execution-layer problem, not a serialization problem. After the plan is deserialized on the shard side, Calcite's `VolcanoPlanner` attempts to re-plan the query and fails to convert the `OpenSearchIndex`-backed table into an enumerable form. This is an architectural issue requiring either:

1. Making `OpenSearchIndex` implement Calcite's `ScannableTable`/`TranslatableTable` interface so the planner can convert it, or
2. Bypassing the planner step on the shard side and executing the deserialized plan directly without re-optimization, or
3. Registering appropriate conversion rules in the shard-side planner context.

The remaining ~164 non-enumerable failures are a mix of: ExchangeLeaf access errors (32), type mismatches (20), lambda/rex issues (16), missing operators (6), and other edge cases.
