# Distributed Query Execution (DQE) Lifecycle вҖ” 3-Node Cluster

**Date:** 2026-04-06

This document traces the complete lifecycle of a SQL query from REST submission to result delivery in a 3-node OpenSearch cluster with Trino split-level distribution enabled.

Each node runs:
- **OpenSearch** вҖ” cluster membership, transport layer, REST endpoint
- **TrinoEngine** вҖ” embedded `DistributedQueryRunner` with `nodeCount=1` (single coordinator per node)
- **OpenSearchSqlTaskManager** вҖ” wraps `SqlTaskManager`, accepts remote task dispatch via transport
- **Trino HTTP server** вҖ” serves exchange pages (data shuffle between stages)

## Initialization Sequence (Node Startup)

### Phase 1: Plugin `createComponents()` (before cluster state)

```
SQLPlugin.createComponents()
  в””в”Җ new TrinoEngine(icebergWarehouse)
       в””в”Җ DistributedQueryRunner.builder(session).setNodeCount(1).build()
            в””в”Җ Starts 1 TestingTrinoServer (coordinator)
            в””в”Җ Installs catalogs: tpch, memory, iceberg, hive
  в””в”Җ trinoEngine.initializeTransportServices()
       в””в”Җ TrinoServiceHolder.initializeWithEngine(engine)
            в””в”Җ Gets SqlTaskManager from coordinator
            в””в”Җ Gets ObjectMapper from coordinator's Guice injector
            в””в”Җ Creates TrinoJsonCodec with Trino-aware serializers
            в””в”Җ Creates OpenSearchSqlTaskManager(sqlTaskManager)
       в””в”Җ Stores Trino HTTP URL (e.g., http://127.0.0.1:38291)
```

At this point, each node can receive and execute transport task requests, but does NOT yet know about other cluster nodes.

### Phase 2: `enableSplitLevelDistribution()` (after cluster state available)

Triggered by the first `ClusterStateListener` callback in `TransportTrinoQueryForwardAction`:

```
ClusterStateListener fires
  в””в”Җ TrinoServiceHolder.enableSplitLevelDistribution(clusterService)
       в””в”Җ TrinoEngine.enableSplitLevelDistribution(clusterService, transportService)
            в”Ӯ
            в”ңв”Җ 1. Create OpenSearchNodeManager(localNode)
            в”Ӯ     в””в”Җ Maps OS DiscoveryNode вҶ’ Trino InternalNode
            в”Ӯ     в””в”Җ Registers local Trino HTTP URL
            в”Ӯ     в””в”Җ Rebuilds activeNodes from cluster state
            в”Ӯ     в””в”Җ Subscribes to ClusterStateListener for future changes
            в”Ӯ
            в”ңв”Җ 2. TransportDistributionPatcher.patch(runner, transportService, nodeManager, codec)
            в”Ӯ     в”ңв”Җ Gets original HttpRemoteTaskFactory from coordinator's Guice injector
            в”Ӯ     в”ңв”Җ Wraps in TransportRemoteTaskFactory(transport, nodeManager, codec, httpDelegate)
            в”Ӯ     в”ңв”Җ Patches SqlQueryExecutionFactory.remoteTaskFactory via reflection
            в”Ӯ     в””в”Җ Patches NodeScheduler вҶ’ NodeSelectorFactory.nodeManager via reflection
            в”Ӯ
            в””в”Җ 3. broadcastTrinoHttpUrl() to all other cluster nodes
                  в””в”Җ Sends TrinoNodeRegisterRequest(myNodeId, myHttpUrl) to each peer
                  в””в”Җ Response includes peer's HTTP URL (bidirectional exchange)
                  в””в”Җ Both sides call nodeManager.registerTrinoHttpUrl()
```

After Phase 2, the coordinator's `NodeScheduler` sees 3 Trino nodes (one per OS node) and can schedule tasks across all of them.

## HTTP URL Exchange Protocol

Each node needs to know every other node's **Trino HTTP URL** (not OS transport address) because data exchange between stages uses Trino's HTTP-based page fetching.

```
Node A starts Phase 2:
  вҶ’ sends trino:node/register(nodeA_id, http://A:38291) to Node B
  вҶҗ receives response(nodeB_id, http://B:41002)

  вҶ’ sends trino:node/register(nodeA_id, http://A:38291) to Node C
  вҶҗ receives response(nodeC_id, http://C:39877)

Result: Node A's OpenSearchNodeManager now has:
  nodeA вҶ’ http://A:38291  (local)
  nodeB вҶ’ http://B:41002  (from register response)
  nodeC вҶ’ http://C:39877  (from register response)
```

`InternalNode` URIs use these HTTP URLs, so `DirectExchangeClient` can fetch pages from any node.

## Query Lifecycle (3-Node Example)

### Example Query

```sql
SELECT regionkey, count(*) FROM tpch.tiny.nation GROUP BY regionkey
```

### Step 1: REST Submission

```
Client
  вҶ’ POST /_plugins/_trino_sql/v1/statement
       Body: "SELECT regionkey, count(*) FROM tpch.tiny.nation GROUP BY regionkey"
       Headers: X-Trino-Catalog: tpch, X-Trino-Schema: tiny
  вҶ’ Arrives at Node A's RestTrinoQueryAction.handlePost()
```

With split-level distribution active, **query-level forwarding is disabled**:
```java
boolean splitLevelActive = TrinoServiceHolder.isInitialized()
    && TrinoServiceHolder.getInstance().getNodeManager() != null;
DiscoveryNode targetNode = splitLevelActive ? null : pickRemoteNode();
// targetNode = null вҶ’ execute locally
```

The query always executes on the node that receives the REST request.

### Step 2: Local Trino Query Planning

```
TrinoEngine.executeAndSerializeJson(sql, catalog, schema)
  в””в”Җ queryRunner.execute(session, sql)
       в””в”Җ Trino's DistributedQueryRunner submits to coordinator
            в””в”Җ SqlQueryManager.createQuery()
                 в””в”Җ SqlQueryExecution (parsing, analysis, optimization, planning)
```

Trino's planner produces a **distributed query plan** with multiple stages:

```
Stage 0 (OUTPUT / root):
  в””в”Җ Partial aggregation result assembly
  в””в”Җ Runs on: coordinator (Node A)

Stage 1 (HASH partitioned):
  в””в”Җ Final aggregation (count)
  в””в”Җ Runs on: any node (scheduled by NodeScheduler)

Stage 2 (SOURCE):
  в””в”Җ Table scan (tpch.tiny.nation) + partial aggregation
  в””в”Җ Runs on: any node (one task per split)
```

### Step 3: Task Scheduling via NodeScheduler

The patched `NodeScheduler` uses `OpenSearchNodeManager` which returns 3 active nodes:

```
NodeScheduler.selectNodes()
  в””в”Җ nodeSelectorFactory.createNodeSelector()  // Uses patched OpenSearchNodeManager
       в””в”Җ Returns {Node A, Node B, Node C}
```

`SqlQueryScheduler` assigns tasks to nodes. For the example:

| Stage | Task     | Assigned Node | Dispatch Method |
|-------|----------|---------------|-----------------|
| 0     | 0.0.0    | Node A        | HTTP (local)    |
| 1     | 0.1.0    | Node B        | Transport       |
| 1     | 0.1.1    | Node C        | Transport       |
| 2     | 0.2.0    | Node A        | HTTP (local)    |
| 2     | 0.2.1    | Node B        | Transport       |
| 2     | 0.2.2    | Node C        | Transport       |

### Step 4: Task Dispatch вҖ” TransportRemoteTaskFactory

```
TransportRemoteTaskFactory.createRemoteTask(session, taskId, node, fragment, splits, ...)
  в”Ӯ
  в”ңв”Җ if node == currentNode (Node A):
  в”Ӯ    в””в”Җ httpDelegate.createRemoteTask(...)    // Standard Trino HttpRemoteTask
  в”Ӯ         в””в”Җ HTTP POST to local Trino HTTP server
  в”Ӯ
  в””в”Җ if node != currentNode (Node B or C):
       в””в”Җ new TransportRemoteTask(transportService, osNode, taskId, node, fragment, ...)
```

### Step 5: TransportRemoteTask вҖ” Remote Task Dispatch

For tasks on Node B or C, `TransportRemoteTask` handles the lifecycle:

```
TransportRemoteTask.start()
  в””в”Җ scheduleSend()
       в””в”Җ STATUS_POLLER.execute(() вҶ’ synchronized(sendLock) { doSendUpdate() })
            в””в”Җ sendUpdate()
                 в”Ӯ
                 в”ңв”Җ Serialize PlanFragment (included until task confirmed created)
                 в”ңв”Җ Build SplitAssignments from pendingSplits
                 в”ңв”Җ Serialize OutputBuffers
                 в”Ӯ
                 в””в”Җ transportService.sendRequest(
                      targetNode,           // OS DiscoveryNode for Node B
                      "trino:task/update",   // Transport action name
                      TrinoTaskUpdateRequest(taskId, fragmentJson, splitsJson, buffersJson)
                    )
  в””в”Җ Start status polling (every 200ms)
```

All sends are serialized through `STATUS_POLLER` under `sendLock` to prevent race conditions:
- The **first send** always includes the `PlanFragment` (the task definition)
- Subsequent sends (addSplits, noMoreSplits, setOutputBuffers) only send deltas
- Fragment is included until a non-PLANNED response confirms the task was created

### Step 6: Worker-Side Task Execution

On the receiving node (B or C):

```
TransportTrinoTaskUpdateAction.doExecute(request)
  в”Ӯ
  в”ңв”Җ TrinoServiceHolder.getInstance().getTaskManager()  // OpenSearchSqlTaskManager
  в”ңв”Җ TrinoServiceHolder.getInstance().getCodec()        // TrinoJsonCodec
  в”Ӯ
  в”ңв”Җ Deserialize PlanFragment, SplitAssignments, OutputBuffers
  в”ңв”Җ Create Session from coordinator's SessionPropertyManager
  в”Ӯ
  в””в”Җ taskManager.updateTask(session, taskId, fragment, splits, buffers, dynamicFilters)
       в””в”Җ SqlTaskManager.updateTask(...)
            в””в”Җ Creates SqlTask with drivers, splits, and pipeline
            в””в”Җ Drivers process splits and produce output pages
```

The worker's `SqlTaskManager` creates the task, starts processing splits, and produces output pages into output buffers.

### Step 7: Data Exchange вҖ” HTTP Page Fetching

Trino's `DirectExchangeClient` on the coordinator fetches output pages from worker tasks via HTTP:

```
Stage 1 reads from Stage 2 outputs:
  DirectExchangeClient
    вҶ’ HTTP GET http://NodeB:41002/v1/task/0.2.1/results/0/0
    вҶҗ SerializedPage (data rows from Node B's table scan)

    вҶ’ HTTP GET http://NodeC:39877/v1/task/0.2.2/results/0/0
    вҶҗ SerializedPage (data rows from Node C's table scan)
```

This is why HTTP URL sharing is critical вҖ” the `InternalNode.getInternalUri()` must point to the Trino HTTP server, not the OS transport address. The URI path format is:

```
{InternalNode.internalUri}/v1/task/{taskId}/results/{bufferId}/{token}
```

The `TransportRemoteTask` constructor sets the self URI correctly:
```java
URI selfUri = URI.create(trinoNode.getInternalUri() + "/v1/task/" + taskId);
```

### Step 8: Status Monitoring

`TransportRemoteTask` polls task status every 200ms:

```
statusPollFuture = STATUS_POLLER.scheduleWithFixedDelay(
    this::pollStatus, 200, 200, TimeUnit.MILLISECONDS);

pollStatus()
  в””в”Җ transportService.sendRequest(targetNode, "trino:task/status", ...)
       в””в”Җ Response: TaskInfo with TaskState (RUNNING, FINISHED, FAILED, etc.)
       в””в”Җ Update lastTaskInfo, lastTaskStatus
       в””в”Җ Notify StateChangeListeners on state change
       в””в”Җ If state.isDone() вҶ’ notify final info listeners, stop polling
```

The coordinator's `SqlQueryScheduler` listens for task state changes to know when stages complete.

### Step 9: Result Assembly

As stages complete bottom-up:
1. **Stage 2** (SOURCE) tasks finish scanning and producing partial aggregations
2. **Stage 1** (HASH) tasks receive shuffled data via exchange, compute final aggregations
3. **Stage 0** (OUTPUT) assembles the result on the coordinator

```
queryRunner.execute(session, sql)
  в””в”Җ Returns MaterializedResult with rows:
       [[0, 5], [1, 5], [2, 5], [3, 5], [4, 5]]
```

### Step 10: REST Response

```
TrinoEngine.serializeToTrinoJson(queryId, result)
  в””в”Җ JSON response with columns, data, stats
  в””в”Җ RestTrinoQueryAction sends BytesRestResponse(OK, "application/json", responseBody)

Client вҶҗ 200 OK
{
  "id": "abc123",
  "columns": [{"name":"regionkey","type":"bigint",...}, {"name":"_col1","type":"bigint",...}],
  "data": [[0,5],[1,5],[2,5],[3,5],[4,5]],
  "stats": {"state":"FINISHED", ...}
}
```

## Transport Actions Summary

| Action Name                    | Direction            | Purpose                                       |
|-------------------------------|----------------------|-----------------------------------------------|
| `trino:task/update`           | Coordinator вҶ’ Worker | Send PlanFragment + splits to create/update task |
| `trino:task/status`           | Coordinator вҶ’ Worker | Poll task state (RUNNING/FINISHED/FAILED)     |
| `trino:task/cancel`           | Coordinator вҶ’ Worker | Cancel a running task                          |
| `trino:node/register`         | Node вҶ” Node          | Exchange Trino HTTP URLs (bidirectional)       |
| `trino:query/forward`         | Node вҶ’ Node          | Query-level forwarding (disabled when DQE active) |
| `trino:task/results`          | (Not used вҖ” HTTP)    | Page fetching uses Trino's native HTTP exchange |

## Key Design Decisions

### 1. Hybrid RemoteTaskFactory (HTTP local + Transport remote)

The coordinator's result pipeline reads the root stage output via Trino's standard HTTP exchange path. If the root stage were dispatched via transport, `DirectExchangeClient` couldn't find it. Therefore:
- **Local tasks** вҶ’ `HttpRemoteTask` (standard Trino path)
- **Remote tasks** вҶ’ `TransportRemoteTask` (OpenSearch transport)

### 2. nodeCount=1 per OpenSearch node

Each OS node runs a single Trino coordinator. Cross-node parallelism comes from split-level distribution (transport), not from multiple Trino workers within one JVM. This avoids memory contention and simplifies the architecture.

### 3. Data exchange via HTTP, not transport

Trino's page serialization format, flow control (backpressure via output buffer ack tokens), and `DirectExchangeClient` are deeply integrated with HTTP. Reimplementing this over transport would require rebuilding Trino's exchange mechanism. Instead, each node's embedded Trino HTTP server handles page serving natively.

### 4. Reflection-based patching

Trino's Guice injector doesn't support overriding bindings after construction. `TransportDistributionPatcher` uses reflection to replace:
- `SqlQueryExecutionFactory.remoteTaskFactory` вҶ’ `TransportRemoteTaskFactory`
- `NodeSelectorFactory.nodeManager` вҶ’ `OpenSearchNodeManager`

This is fragile but necessary because `DistributedQueryRunner.builder().setAdditionalModule()` adds to the module list rather than replacing bindings.

### 5. Serialized sends via STATUS_POLLER

All `TransportRemoteTask` sends go through a single-threaded executor under `sendLock`. This prevents race conditions where `addSplits()` could send before `start()` completes, which would cause "fragment must be present" errors on the worker.
