## Analysis

### Why Trino-OS is 2x slower

1. **~4s per-query overhead floor**: Every query pays REST API + query parsing + planning cost. Spark's warm cache runs are sub-1s for simple queries. This explains the 5x gap on simple scans (Q0-Q3) where the actual computation is trivial.

2. **Single run vs best-of-3**: Spark results use the best (minimum) of 3 runs, benefiting from OS page cache warmup. Trino-OS results are single-run cold.

3. **Embedded overhead**: Trino runs inside OpenSearch's JVM via a 400MB shadow jar with 40+ package relocations. Standalone Trino/Spark avoids this overhead.

4. **Memory configuration**: Trino-OS uses 10GB query memory within 16GB heap. Spark has 32GB RAM available.

### Where Trino-OS performs well

- **Q23 (0.4x ratio)**: `SELECT * WHERE URL LIKE ORDER BY LIMIT` — Trino's TopN + predicate pushdown is faster than Spark's full sort.
- **Complex queries (1.6x)**: On queries with complex predicates, JOINs, or HAVING clauses, the gap narrows because Trino's cost-based optimizer is competitive.
- **Q22 (1.4x)**: Multi-predicate filter with DISTINCT — close to Spark performance.

### Optimization opportunities

1. **Reduce per-query overhead**: Pre-warm the query planner, cache parsed plans
2. **Run best-of-3**: Would reduce times by ~20-30% with warm OS page cache
3. **Increase parallelism**: Currently single-node with `setNodeCount(1)`
4. **Use Iceberg instead of Hive**: Iceberg metadata provides column stats for predicate pushdown
5. **Native Parquet reader**: Trino's Parquet reader may not be as optimized as Spark's
